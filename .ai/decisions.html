ADR-001 — Webcam Studio: single-file browser app

Date: 2026-07-17 · Status: Accepted · Project: webcam-recorder/

Context

User wanted an application to record their webcam and produce a high-quality video clip on macOS.

Decision

Build a zero-dependency, single-file browser application (index.html) using getUserMedia for capture and MediaRecorder for encoding. No install, no build step, no server beyond optional python3 -m http.server for Safari.

Alternatives considered

OptionVerdictWhy
Python + OpenCV/PyAVRejectedDependency friction, audio/video sync is DIY, encoding quality harder to control.
Electron wrapperRejectedHundreds of MB to deliver what the browser already does natively.
Native Swift/AVFoundationDeferredBest pipeline control, but needs Xcode build/signing; overkill for v1. Revisit if the app grows (virtual camera, editing, streaming).
Browser appChosenHardware-accelerated encode for free, permission model built in, portable, immediately runnable.

Codec & quality strategy

  • MIME preference order: MP4 H.264 High + AAC → plain video/mp4 → WebM VP9/Opus → VP8/Opus → WebM. First entry passing MediaRecorder.isTypeSupported wins. MP4-first because the user is on macOS (QuickTime/Quick Look/iMovie).
  • Bitrate presets 8/14/24 Mbps at 1080p, scaled linearly by pixel count, clamped to 4–80 Mbps. Audio 192/256 kbps with echoCancellation, noiseSuppression, autoGainControl.
  • recorder.start(1000) timeslice so chunks flush progressively (live file-size readout, less loss on crash); falls back to start() where timeslice is unsupported.
  • Preview is mirrored (cosmetic); recordings are never mirrored.

Known limits

  • Safari may refuse camera on file:// — README documents the localhost workaround.
  • WebM fallback does not play in QuickTime; README gives the one-line ffmpeg conversion.
  • No editing/trimming in-app — record, review, re-record is the workflow.

ADR-002 — Screen share, teleprompter, persistence (v2)

Date: 2026-07-17 · Status: Accepted · Supersedes: extends ADR-001

Context

Requested: optional screen-share recording with toggleable webcam, an optional teleprompter that follows the speaker's voice (highlight next N words, default 5), adjustable prompter pace, and persistence of last-used settings (including the script) across sessions.

Decisions

  • Split mic and camera into independent getUserMedia streams. The webcam can then be stopped entirely during screen share (camera LED off = real privacy) while the mic keeps recording. Camera failure no longer kills audio.
  • Screen + webcam PiP via a hidden canvas compositor (canvas.captureStream), so the recorded file matches the preview exactly (WYSIWYG). Screen-only and webcam-only modes record tracks directly with no canvas in the path — zero compositing overhead in the common cases.
  • Screen audio mixed via WebAudio (MediaStreamDestination) only when both mic and shared-tab audio exist — MediaRecorder encodes a single audio track. macOS can only capture tab audio; documented in README.
  • Teleprompter is a DOM overlay, never drawn to the recording canvas. Speech follow uses the Web Speech API (SpeechRecognition/webkitSpeechRecognition, continuous + interim results). A monotonic word cursor advances by matching recognized words within a lookahead window (12 words for final results, 6 for interim, prefix-matching for interim ≥3 chars). No backward movement — prompters only go forward.
  • Two follow modes: speech-follow (default) and explicit timed scroll (60–220 wpm). Timed is also the automatic fallback when recognition is unavailable/blocked/offline (Firefox, no network, denied). Pace, font size, highlight count and mode stay adjustable mid-recording; only the script textarea locks while rolling. ↑/↓ keys bump pace live.
  • Persistence via localStorage (key webcamStudio.settings.v2) rather than cookies: no server round-trips, no expiry, larger quota. Device IDs are re-validated against enumerateDevices before reuse (stale IDs are dropped). Screen-share source is not restored on boot — getDisplayMedia requires a fresh user gesture each session.

Known limits (v2)

  • Chrome/Edge speech recognition sends audio to the vendor's cloud service (needs internet); Safari uses its own service. Firefox has no Web Speech API → timed scroll. All paths degrade gracefully and say so in the UI.
  • Speech recognition listens on the system default mic, not the app-selected one.
  • If the user shares the screen/window containing the app itself, the prompter overlay is visible in the capture — inherent to screen recording; README advises sharing a different window/tab.
  • Canvas PiP caps at the shared screen's native resolution; webcam inset is fixed at 25% width, four corners.

ADR-003 — Burned-in watermark via unified canvas compositor

Date: 2026-07-17 · Status: Accepted · Extends ADR-002

Context

Requested: a very small credit line across the bottom of the video, part of the recording (default Webcam recording Agent: copyright Sahasi <current year>), user-editable.

Decision

  • Text can only enter the recorded file by drawing it onto frames, so when a watermark is set, all modes route through the canvas compositor (generalized from PiP-only to: any primary source + optional webcam inset + watermark). Preview shows the canvas, keeping WYSIWYG.
  • Empty watermark field ⇒ direct-track recording (zero canvas overhead). The blank-is-off convention avoids a separate toggle.
  • Default string built with new Date().getFullYear() at boot; a saved value (even intentionally blank) overrides the default. Persisted in localStorage like all other settings.
  • Font scales with frame height (~1.6%, ≈17px at 1080p), auto-shrinks to fit 92% of frame width, semi-transparent white with a soft shadow for legibility on light content. Centered, bottom margin ≈1.6× font size.

Trade-offs accepted

  • Canvas painting is throttled in background tabs — with a watermark set, the tab should stay visible while recording (documented). Direct-track mode (no watermark) is unaffected.
  • Mirror preview is disabled while a watermark is active: CSS mirroring would flip the text too, breaking preview-equals-recording.

ADR-004 — Teleprompter follow engine v2 (two cursors + pace extrapolation)

Date: 2026-07-17 · Status: Accepted · Supersedes: the single-cursor design in ADR-002

Context

User feedback: (1) the highlight visibly skipped words, which was disorienting; (2) words were greyed out as "spoken" when they had not been spoken — the grey boundary must only advance on certainty.

Decision

  • Two cursors. confirmedIndex (grey boundary) advances only on final speech results matched within 5 words. focusIndex (highlight) is speculative: interim results and far final matches (up to 10 words) may move it, but never grey anything. Single-letter words are ignored as recognition noise.
  • Pace extrapolation. A 250 ms engine tick measures the reader's pace from the last ≤12 final-match samples (clamped 0.6–5 words/s, slider value as cold-start default). After 600 ms of recognition silence, the highlight glides forward fractionally at that pace; capped at +10 words past the last speech anchor and suspended after 6 s of silence (reader paused). New matches re-anchor; the highlight never moves backward.
  • Timed-scroll mode drives both cursors together (it asserts "said" by definition).
  • Verified by a 41-case node test suite driving the real script with a stub DOM and a controllable clock.

ADR-005 — Dual license + stealth watermark editing

Date: 2026-07-17 · Status: Accepted

Context

Requested: a licensing model that requires retaining the copyright/watermark message unless a paid version is used; the watermark textbox must not be editable unless the user clicks its label, with no hover-cursor hint.

Decision

  • Dual license in LICENSE: (A) Free Attribution License — MIT-style permissions conditioned on retaining the copyright notice, the watermark feature, and its default attribution text in all copies/deployments/recordings, plus no-sale; (B) paid Commercial License waiving those conditions. No OSI-approved license supports payment-conditional attribution, so clause (A)(2) is a custom source-available restriction; the software is not "open source" in the OSI sense under this model.
  • Stealth edit affordance. The watermark input is readonly by default; clicking the label removes the attribute (native label behaviour then focuses the field — no custom cursor styling anywhere, so nothing hints at clickability). Re-locks on reload. Documented in README, since a license-required control must still be discoverable by legitimate users.

ADR-006 — UI refactor: broadcast-equipment language + progressive disclosure

Date: 2026-07-17 · Status: Accepted · Method: frontend-design skill (subject grounding → token system → anti-default critique)

Context

Requested: refactor and clean up the UI. Explicit user steer: "not just colors — icons, grouping things intuitively, hiding complexity, keeping it simple." The v1 UI was two always-visible fieldsets (~19 controls) plus unicode-glyph buttons: a wall of settings for a task whose core loop is see yourself → record → stop → download.

Decisions

  • Progressive disclosure via native <details>. Default surface = monitor + transport + a 5-icon quick row. Teleprompter and Capture settings collapse into disclosure panels (no JS needed for open/close; keyboard-accessible for free). Settings persist across sessions, so hiding them post-setup costs nothing.
  • Icon quick row for per-take toggles: segmented webcam/screen source switch, mic mute, teleprompter, settings gear — inline SVGs (no emoji, no unicode ● ■ ⏸ approximations). Each icon proxies to the canonical settings input (btnPrompterToggle flips prompterEnable and fires its handler), so persistence and every existing handler work unchanged. syncQuickUI() re-derives icon states from ground truth (mode, micMuted, prompterOn) after every transition.
  • Single source of truth per control. Mic mute moved out of settings into the quick row (state var micMuted, was a second checkbox). Mic level moved out of the settings grid onto the monitor as an analog VU needle — the design's signature element (fast attack / slow decay ballistics, vuAngle += (target − vuAngle) × (0.5 up / 0.12 down)).
  • Broadcast-equipment visual language. Tokens: warm charcoal #141210, brass #d9a05b, tally red #e2483d, VU cream; Barlow Condensed (labels) / Barlow (body) / IBM Plex Mono (readouts). Supporting chrome: viewfinder corner brackets (hidden in review playback), tally-lamp status pill, deck-style transport with real SVG glyphs.
  • Critique pass applied: rejected bezel screws and numbered VU markers as costumery; kept only elements that carry state (needle = level, tally = status, corners = live-vs-review framing).

Stability guarantees

  • All element IDs the 41-case node test suite drives are preserved; harness stub gained only setAttribute. Suite passes 41/41 after the refactor.
  • Source switching via icons is inert during countdown/recording/paused; mute and prompter toggles stay live mid-recording by design.
